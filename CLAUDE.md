## The Rules

1. **Work = it runs.** A file is not done because it looks right. It is done when it demonstrably works. Prioritize correctness over elegance.
2. **One file at a time.** Write it, commit it, wait for confirmation it works. Only then move to the next file.
3. **No scope creep.** Do not add features, buttons, logic, or UI elements that were not explicitly agreed on. If an idea comes up mid-build, note it and park it.
4. **No rewrites without a branch.** Large changes go on a branch named `refactor-<subject>`. Small targeted fixes go directly to `main`.
5. **Test after confirm.** The user confirms a feature works manually first. Then a test is written to lock that behavior in permanently. Tests are never written speculatively.
6. **Fail loudly.** Scripts that produce no output or encounter errors must exit with a non-zero code. Silent failures are not acceptable.
7. **One approval request at a time.** Do not ask follow-up questions or propose the next step until the current one is confirmed done.

```
PotholeDodgerV2/
├── index.html                        ← redirect shim → web/index.html
├── scripts/
│   └── rva311_pipeline.py            ← fetches RVA 311 data, outputs web/data/hazards.json
├── web/
│   ├── index.html                    ← main app UI
│   ├── css/
│   │   └── theme.css
│   ├── js/
│   │   ├── geocode.js                ← address → {lat, lng}
│   │   ├── routing.js                ← OSRM route → {polyline, streetNames}
│   │   ├── hazards.js                ← load hazards.json, filter by street name
│   │   ├── map.js                    ← Leaflet map, draw route, plot markers
│   │   └── app.js                    ← wire everything together
│   └── data/
│       └── hazards.json              ← generated by pipeline, NOT hand-edited
├── tests/
│   └── (test files go here)
└── .github/
    └── workflows/
        └── run-pipeline.yml          ← runs rva311_pipeline.py on workflow_dispatch or pipeline push
```

### Data flow
1. User enters origin + destination
2. `geocode.js` resolves both to `{lat, lng}`
3. `routing.js` fetches OSRM route → returns `{polyline, streetNames[]}`
4. `hazards.js` loads `web/data/hazards.json`, filters by street name match against `streetNames[]`
5. `map.js` draws the route polyline and plots matching hazard markers on a Leaflet map
6. User sees the route and all hazards on it

### External dependencies
- **OSRM** — open source routing, no API key required
- **Leaflet** — map rendering
- **Nominatim** — geocoding (via OpenStreetMap), no API key required
- **RVA 311 API** — `POST https://webapi.citizenservices.org/rvaone/api/v1/requests` — no auth required

## Hazard Data

- Source: RVA 311 (`webapi.citizenservices.org`)
- Refreshed: manually via GitHub Actions `workflow_dispatch`, or automatically on push to pipeline script
- Output: `web/data/hazards.json`
- Schema per record:
```json
{
  "id": "DPW000231648",
  "serviceName": "Pothole on Road",
  "latitude": 37.5418839,
  "longitude": -77.4462294,
  "location": "8 1/2 W Canal St, 23220",
  "status": 2,
  "requestDate": 1775146620700,
  "description": "..."
}
```
- `status` is a number as returned by the API
- `requestDate` is a Unix timestamp in milliseconds
- 766 records as of 2026-04-02

## Street Name Matching

Potholes are matched to a route by comparing the pothole `location` field against the street names returned by OSRM.

**Algorithm (in `hazards.js`):**
1. Split pothole `location` on `&` to handle intersections (gives 1–2 tokens)
2. Expand abbreviations on both sides: `St→Street`, `Ave→Avenue`, `Blvd→Boulevard`, `Dr→Drive`, `Rd→Road`, `Pkwy→Parkway`, `Ln→Lane`, `Ct→Court`, `W→West`, `E→East`, `N→North`, `S→South`
3. Strip house numbers (leading digits) and punctuation
4. Normalize to lowercase
5. Check if any route street name **contains** or **is contained by** any normalized pothole token

This normalization function must be pure and tested in isolation.
